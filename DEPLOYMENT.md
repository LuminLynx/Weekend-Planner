## üöÄ Alternative Backend Deployment Options

While the **live static app runs without any backend**, if you need to deploy the full Python FastAPI backend for custom integrations or real API connections, consider these options:

### Option 1: Railway (Recommended - Free Tier Available)

**Pros:** Easy deployment, free tier, automatic HTTPS, great for hobby projects  
**Deployment Steps:**

1. Create account at [railway.app](https://railway.app)
2. Click "New Project" ‚Üí "Deploy from GitHub repo"
3. Select the Weekend-Planner repository
4. Railway auto-detects the Dockerfile
5. Your API will be live at: `https://your-app.railway.app`

**Configuration:**
- No additional configuration needed
- Railway uses the existing `Dockerfile`
- Automatic HTTPS and custom domains

### Option 2: Render

**Pros:** Free tier, easy setup, automatic deploys from GitHub  
**Deployment Steps:**

1. Create account at [render.com](https://render.com)
2. Click "New Web Service"
3. Connect your GitHub repository
4. Render auto-detects Python
5. Use these settings:
   - **Build Command:** `pip install -e .`
   - **Start Command:** `uvicorn app.server:app --host 0.0.0.0 --port $PORT`

**Configuration:**
```yaml
# Create render.yaml in repository root
services:
  - type: web
    name: weekend-planner
    env: python
    buildCommand: pip install -e .
    startCommand: uvicorn app.server:app --host 0.0.0.0 --port $PORT
    healthCheckPath: /healthz
```

### Option 3: Fly.io

**Pros:** Global edge deployment, good free tier, great performance  
**Deployment Steps:**

1. Install flyctl: `curl -L https://fly.io/install.sh | sh`
2. Login: `fly auth login`
3. Initialize app: `fly launch`
4. Deploy: `fly deploy`

**Configuration:**
```toml
# fly.toml (generated by fly launch)
app = "weekend-planner"

[build]
  dockerfile = "Dockerfile"

[env]
  PORT = "8000"

[[services]]
  http_checks = []
  internal_port = 8000
  processes = ["app"]
  protocol = "tcp"

  [[services.ports]]
    port = 80
    handlers = ["http"]

  [[services.ports]]
    port = 443
    handlers = ["tls", "http"]
```

### Option 4: Heroku

**Pros:** Well-established, many add-ons available  
**Note:** Free tier no longer available, paid plans start at $5/month

**Deployment Steps:**

1. Install Heroku CLI: `npm install -g heroku`
2. Login: `heroku login`
3. Create app: `heroku create weekend-planner`
4. Deploy: `git push heroku main`

**Configuration:**
```
# Procfile
web: uvicorn app.server:app --host 0.0.0.0 --port $PORT
```

### Option 5: AWS Lambda + API Gateway (Serverless)

**Pros:** Pay-per-use, scales automatically, AWS free tier  
**Deployment Steps:**

Use [Mangum](https://github.com/jordaneremieff/mangum) to wrap FastAPI:

```python
# lambda_handler.py
from mangum import Mangum
from app.server import app

handler = Mangum(app)
```

Deploy using AWS SAM or Serverless Framework.

### Option 6: Google Cloud Run

**Pros:** Serverless containers, pay-per-use, free tier  
**Deployment Steps:**

1. Install gcloud CLI
2. Build: `gcloud builds submit --tag gcr.io/PROJECT-ID/weekend-planner`
3. Deploy: `gcloud run deploy --image gcr.io/PROJECT-ID/weekend-planner --platform managed`

## üîí Production Considerations

If deploying the Python backend to production:

### Security
- [ ] Add authentication/API keys if needed
- [ ] Set up rate limiting
- [ ] Use HTTPS (most platforms provide this automatically)
- [ ] Validate all user inputs
- [ ] Set CORS policies appropriately

### Environment Variables
```bash
# .env file (never commit this)
API_KEY_VENDOR_A=your_key_here
API_KEY_VENDOR_B=your_key_here
FX_API_KEY=your_fx_api_key
DINING_API_KEY=your_dining_api_key
```

### Monitoring
- Enable platform logging (all platforms provide this)
- Set up uptime monitoring (UptimeRobot, Pingdom)
- Monitor API usage and costs
- Track error rates

### Performance
- Enable platform caching where available
- Use CDN for static assets
- Implement request caching
- Set appropriate timeout values
